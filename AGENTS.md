# Agent rules

## Git: mutation policy (hard)

### Remote mutation — never

**Never mutate remotes.** This holds even if the user explicitly asks.

Forbidden to execute (non-exhaustive):

- `git push` / `git push --force` / `git push --delete`
- Creating, updating, merging, or closing PRs/issues via CLI or API
- Remote branch/tag create/delete/move
- Publishing releases, uploading artifacts, or changing remote repo settings
- Any other write to GitHub/GitLab/etc.

If remote mutation is needed: **print the exact commands** (and any `gh`/`git` args) for the user to run. Do not run them.

### Local mutation — permission required

Local writes that change git state need an explicit ask first, then execute only what was approved:

- `git commit`, `git add` / unstage, branch create/switch/delete
- `git merge`, `rebase`, `cherry-pick`, `reset`, `stash` (apply/drop)
- Amending commits, rewriting local history, deleting local branches

When committing: stage only the paths relevant to the request; no drive-by files.

### Read — unrestricted

Read anything freely, local or remote:

- `git status`, `diff`, `log`, `show`, `blame`
- `git fetch` is allowed only if it does not change local branch tips in a surprising way; prefer pure reads (`git log origin/...`, `gh pr view`, API GET). If fetch is needed to refresh remotes, say so and get permission first when it would update refs the user cares about — when in doubt, ask or give the command.
- Remote **reads** (`gh pr view`, browse PR diffs, clone metadata) are fine

Default: **read freely; change local git only when asked; never change remote git yourself.**

## Commit messages

When asked to commit or draft a message, write a **clean, brief, information-dense** message from the **actual diff** (and changelog/docs when they explain intent). No ads, “Generated by …”, co-author trailers, or emoji signatures.

### Do not copy human commit style

Past human commit messages in this repo are **not** the quality bar. On average they are worse than a careful LLM message. Use history only for:

- **Conventions that are structural** (e.g. whether the repo uses `feat:` prefixes, ticket IDs, or version tags)
- **Facts** (what landed when), not tone or fluff

Do **not** mirror vague subjects (“fix stuff”, “updates”, “wip”) or under-specified one-liners just because `git log` is full of them.

### How to research

1. `git status` + full relevant `git diff` (staged and unstaged) — source of truth  
2. `CHANGELOG.md` / version notes / nearby design docs when they state user-facing intent  
3. `git log` lightly — structure/conventions only, not prose inspiration  

### Message shape

- Imperative subject, ≤ ~72 chars: “Add…”, “Fix…”, “Remove…”, not “Added…” / “Fixes…”  
- Conventional commits (`feat:`, `fix:`, …) **only if** the repo already uses them consistently for automation; otherwise plain imperative  
- Body only when needed: 1–3 short lines on **why**, user-visible impact, or non-obvious tradeoffs  
- No file lists, no “this commit…”, no filler  

### Content

- Prefer **behavior and motivation** over hunk inventory  
- One logical change per commit when practical; if the diff mixes concerns, say so or split if asked  
- Never invent scope that is not in the diff  

### Examples

**Good**

```
Fix race where session refresh overwrote in-flight cart updates

Refresh merges server cart by line id instead of replacing the
whole cart, so concurrent add-to-cart no longer drops items.
```

**Bad**

```
Update files and fix stuff

- modified cart.js
- improved code quality
```

### Mid-chat shorthand

> Draft/commit message for current changes: brief, imperative, dense; why over file lists; no AI trailer. Diff is source of truth; don’t ape weak history prose.
