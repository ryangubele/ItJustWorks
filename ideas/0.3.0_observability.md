# 0.3.0 - Observability: make it show its work

**Grounded in:** *Make it observable* - now a stated **practice** in [`DESIGN.md`](../DESIGN.md), standing on five principles at once: respect the player, respect the contributor, never break a save, modern habits, and no ego (a tool that can't watch itself is claiming it never errs). Also the hard prerequisite for 0.5.0: a smarter detector is only as trustworthy as your ability to watch it decide.

## A fix that leaves no trace

0.2.0 shipped a save-safety fix - `ElapsedInScene` re-baselines the stuck-timer after a full restart, so a scene stuck across a relaunch still alerts instead of going silent forever (the S-0 case). It works. But the re-baseline branch writes **nothing** to the log; trace on or trace off, identical. So the one correction that matters most, on the one path that's hardest to reproduce, leaves no evidence it fired - you can only infer it backwards from the alert eventually going off. A fix you can't watch fire is a fix you can't verify, and this one guards the prime directive. That isn't a one-off; it's a pattern - the tool makes silent corrections.

## What's currently invisible

Every self-heal is mute unless it happens to route through a `Trace` we already had:

- `ElapsedInScene` re-baseline after a real-time reset (S-0) - silent.
- `EnsureHist` rebuilding a history ring resized by another build - silent.
- `PlayerRef` recovery in `OnInit` when the VMAD fill came up empty - silent.
- `ReassertRegistrations` re-registering a dropped key/timer on MCM open - silent; it just calls the registers.
- The poll re-arming itself each tick - silent by design, but there's no cheap way to confirm the loop is even alive.

When one of these quietly saves a run, we learn nothing. When one fails to, we have no breadcrumb to start from.

## The cost (why this is a contributor issue, not a nicety)

Anything scene-related has to be verified on a real, heavy load order - that's where scenes actually break. If confirming a fix means firing up a 500-plugin game, coaxing out a flaky stuck-state, and doing it over and over because the tool won't say what it saw or did, the barrier to contributing - or to *us* testing before a release - is brutal. Good logging is the whole difference between one careful reproduction and sixteen frustrated ones. That's respect for whoever's time is on the line, ours included.

## The direction

- **Trace every correction and every state transition, not only the ones already instrumented.** If the tool changes its own state or heals something, it says so, at a consistent level. A thing that matters and happens quietly is a bug in the telling.
- **Greppable, structured lines.** The stable `[fth_IJW]` prefix plus a consistent shape - event, scene, values - so a log is filterable, not prose. Probably a subsystem tag (poll / heal / alert / hotkey).
- **Levels, not one boolean.** `bTrace` is all-or-nothing today. At least off / normal (corrections + transitions) / verbose (every poll), and cheap when off.
- **More state in the readout, too.** The MCM already mirrors live state; extend it where it lets a player self-diagnose without ever opening a log (last correction, last alert, a loop-alive indicator).
- **Document the pipe.** `bTrace` writes nothing unless Papyrus logging is enabled in the ini. A tool that claims observability has to tell you how to turn the tap on - path and ini flags, in the manual.

## Why it must precede 0.5.0

The 0.5.0 detector is a judgment machine - two clocks, a relaxed backstop, per-scene thresholds, a long-scene list. Tuning any of it means answering "which signal tripped, at what value, on which scene, and was that right?" - over and over, on real load orders. Ship the detector first and you tune blind: false positives and misses collapse into indistinguishable folklore. Observability is the instrument that lets you *earn* the detector. First the gauge, then the smarter engine.

## Open questions

- Verbosity model: a single enum, or per-subsystem flags? What that looks like in the MCM.
- Division of labor: what belongs in the readout a player sees vs. the log a debugger reads?
- A structured line format worth committing to *before* 0.5.0 starts emitting detector decisions into it.
- Performance floor: `Trace` is cheap, but a verbose per-poll line on a fast poll adds up over a long session - does verbose need a guard?
- How much ships user-visible vs. staying under the Diagnostics hood until 0.5.0 needs it.
