# 0.5.0 - Detection: better accuracy, less noise

**Sequencing:** this is the hardest idea to get right and the hardest to verify, so it sits behind the two releases that make it *possible*. 0.3.0 (observability) has to land first - a two-clock detector with a relaxed backstop and a long-scene list is untestable without being able to see which signal tripped and when, on a real load order. 0.4.0 (the joke toggle) is independent and rides ahead of it. Detection is targeted at 0.5.0 not because it's low priority - it's the whole point - but because shipping it blind would violate the prime directive. Build it when the ground under it is ready.

## The question

How do we decide a scene has been active "too long" and is probably stuck? Today it's one signal - wallclock real-time, default 3 min (`Watcher.psc`: `fSceneFirstSeen` from `GetCurrentRealTime`, alert when `elapsed > fAlertThreshold`). That's the *accessible* signal, not the best one.

## Two clocks, complementary (not redundant)

Redundant would be two ways to measure the same quantity. These measure *different* quantities and catch *disjoint* failure modes.

- **Wallclock** (`GetCurrentRealTime`): real seconds. Accrues while you're AFK, including in a paused menu. Resets per launch (the S-0 headache, fixed in 0.2.0 via re-baseline). Intuitive to message.
- **Game-time** (`GetCurrentGameTime`): the in-world calendar (days, float). Freezes on menu pause. Jumps on sleep/wait. Persists in the save (S-0-immune). Timescale-independent as a threshold.

| Situation | Wallclock | Game-time |
|---|---|---|
| AFK in a menu (benign) | **false alarm** (accrues) | correct (frozen) |
| Slept mid-scene, scene BROKE | misses / delays (~1 real min) | **catches** (jumps) |
| Slept mid-scene, scene FINE | correct | false alarm |
| Stuck, world running (common) | catches | catches |
| Stuck, world paused / menu-locked | **catches** | misses (frozen) |
| Long *legit* scene, world running | false alarm if long | rarely (needs game-hours) |

Game-time is the more *semantic* measure of "how long has this scene been the state of the world" - which is what stuck means - and it's S-0-immune. Wallclock's unique catch is paused-world-stuck (rarer). Sleep-induced breakage is a real class only game-time sees.

## The direction

**A two-signal detector: alert if `(wallclock > T_real)` OR `(game-time > T_game)`.**

- Union of detectors is a strictly better watchdog; bias toward not-missing.
- Adding game-time lets us *relax* wallclock. Today's aggressive 3 min becomes a conservative backstop (~20 min), while game-time catches sleep / fast-forward / long-active. Lowers false alarms AND raises coverage - the rare change that improves both at once.
- The alert is advisory (a dismissable notification, not an auto-Stop), so a false positive is low-harm; the failure that kills saves is a MISS. Err toward annoying.
- The message should name which signal tripped - the game-time phrasing ("~3 in-game days") is a stronger "this is wrong" cue than "~3m".
- If ever forced to one signal, game-time edges wallclock: rarer false positive, S-0-immune, more of the failure surface - at the cost of missing paused-world-stuck and being harder to phrase.

## Two related noise problems to fix in the same pass

### Alerts missed under a menu pause (confirmed)

The poll fires during a menu pause (test logs showed notifications missed while in a menu). A `Debug.Notification` shown while the player is in inventory / map / etc. can be missed entirely. This is also the real reason 3 min *feels* too aggressive: it isn't over-eager detection - it's that inventory-management-mid-scene (a real Skyrim habit; even the short Calixto tour trips it) both accrues wallclock AND hides the alert.

Fix idea: a periodic *re-alert* while still stuck (wallclock-cadenced is fine), so someone lingering in a menu doesn't miss the one-shot. Re-alert until the scene clears or the player acknowledges - today `bAlerted` is one-shot per scene.

### Long legit scenes will false-alarm (pernicious)

The longest vanilla scenes (Season Unending / the peace council) run long enough to trip any *useful* threshold. We can never know from clock time how long a mod intends a scene to run.

Not a pure whitelist ("ignore this scene") - long scenes can still get stuck. Even Season Unending (which uses `ForceStart`) can leave a player stopped mid-way or stuck on its final scene. Instead: a **long-scene list** - known-long scenes get a *longer threshold*, not an exemption. Still watched, just with a higher bar. Needs a data source, a maintenance story, and eventually a way for mods to register their own long scenes.

## Stance

Some false positives are unavoidable - no clock can read a mod author's intent. Two competing duties: minimize false positives, and don't fire so late the save is already broken. Always a judgment call. Err toward annoying: a dismissable false notification is cheap; a missed stuck scene is the save-killer this exists to prevent. But the current 3 min is too aggressive - chiefly the menu-pause artifact, not detection eagerness.

## Open questions

- Default thresholds `T_real` / `T_game` - empirical; test against real load orders. Starting guess ~20 min real / ~5 game-hours (5 game-hrs is roughly 15 real-min at default 20x while the world runs; instant via sleep).
- Re-alert cadence and acknowledgment mechanism; its own tunable?
- Long-scene list: how to source and maintain it, per-scene threshold override, MCM surface, third-party registration.
- Default posture: both signals on by default? Both tunable (incl. 0 = off) like today's poll/warn?
- In-game verification: confirm poll-during-pause precisely; `GetCurrentGameTime` behavior under sleep / wait / pause; timescale independence.
- Interaction with the shipped S-0 re-baseline (keep it for the wallclock half; the game-time half needs none).

## For context: S-0 (fixed in 0.2.0)

Wallclock is session-relative and resets each launch, but `fSceneFirstSeen` persists in the save - so after a reload while still stuck, `elapsed` went negative and the alert never fired. Fixed with a re-baseline (`ElapsedInScene`) that restarts the timer from the reload. Note: grounding on game-time would have sidestepped S-0 entirely - the game-time instinct had that merit too.
