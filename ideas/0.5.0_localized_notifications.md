# 0.5.0 - Localized notifications: speak the player's language

**Grounded in:** *Respect the player* - an advisory the player can't read is half an advisory - **and** *Respect the contributor* - toast copy lives in the same tables translators already edit.

**Sequencing:** Lands **before detection** so functional toast rails (tables → bake → language index) exist before multi-clock alert copy multiplies. Design is closed enough to implement; not blocked on auto-detect, runtime table parse, or `$key` into `Debug.Notification` (that path does not resolve; do not pursue it).

---

## The goal

Functional toasts - stuck-scene alert, "names are off" notice, Stop results - print English today while the MCM and manuals are localized. Close that corner: those lines speak the player's language.

The punchline `See? It Just Works!` stays English on purpose: the joke is English-native. That is taste, not an engine limit.

---

## Why a setting (not game-language auto-detect)

Toast language should follow **menu intent**, not `sLanguage`.

This mod's installer (and the by-hand rename in the manuals) can put Chinese, German, etc. **content** into `fth_ItJustWorks_ENGLISH.txt` while the game stays English. Auto-detecting game language would leave English toasts for the players who already localized the menu - the ones who care most.

So language is an ordinary **ModSettingInt** (same channel as poll interval and log level): overridable, resettable, independent of the save.

Secondary fact: SkyUI resolves `$keys` for menu display only. There is no free Papyrus path that turns `$fth_IJW_…` into a string for `Debug.Notification`. The script needs a real string; the setting picks which baked string.

A control the player can set is not a half-measure. If the menu is already in their language, Settings is findable. Default English preserves today's behavior; installer seed (below) makes the common path zero-click.

---

## Design - three parts, all in the existing stack

**1. Source of truth: the translation tables.**  
Toast strings live in `interface/translations/fth_ItJustWorks_<LANG>.txt` next to every other string. No second file format, no second contributor path.

**2. Build bakes them into the script.**  
The build already reads all ten tables (key-parity gate) and compiles Papyrus. It generates a lookup such as `ToastText(index)` (or per-key helpers) from a marked set of toast keys. Compile-time only: no runtime file I/O, no new dependency, nothing on the poll hot path. Fix Spanish in the table → next build fixes the Spanish toast.

**3. MCM enum picks the index.**  
"Notification language" (working name) - `sourceType: ModSettingInt`, same pattern as log level. Labels are `$keys`; value is an index. Alert line: `ToastText(index) + ElapsedLabel(...)` with language-neutral `~3m` / `~90s`. Punchline stays a separate English literal, kept out of the bake - English by taste, and the shipped Levity toggle already gates it.

## Installer seed (Config defaults, not user Settings)

MCM Helper load order:

1. **`Data/MCM/Config/fth_ItJustWorks/settings.ini`** - author defaults (shipped; FOMOD may rewrite this file at install).
2. **`Data/MCM/Settings/fth_ItJustWorks.ini`** - player overrides only for keys they changed from default.

User file is created when a setting **leaves** its default. New keys that never appear there fall through to Config. That is intentional and upgrade-safe.

**Seed path:** the same install step that already chooses default menu language also writes `iToastLang` (name TBD) into **Config** `settings.ini` to match. English remains index 0 in the package baseline; non-English default-menu options patch the default value when selected.

**Do not** seed by writing `MCM/Settings/` - that would plant a permanent user override and fight "reset to default."

**Upgrade:** an existing player has a Settings file without the new key. Config gains the key (and, if they re-run FOMOD and pick a default menu language, the matching seed). Missing user key → Config value. No special migration script.

**Soft edge:** a silent update that never re-runs the FOMOD language step leaves Config at package English default; menu file may still be non-English from an older install. Enum still fixes it in one click. Same class of edge as any installer-only default.

English (0) is the fallback and the reset target when Config says English.

## Scope

Functional toasts only:

| Line | Notes |
|------|--------|
| Stuck-scene alert prefix | First-class; mid-crisis; + neutral elapsed |
| Names-off notice | Same rails; low traffic |
| Stop success / fail | Same rails; low traffic |

Hotkey readouts can stay English or join later cheaply once the bake exists. Punchline: English only.

Later multi-clock copy (detection: game-time vs wallclock phrasing) reuses the same rails; new prefixes are more table keys, not a second pipeline.

---

## Cost

Near nothing the player would notice: no new dependency, no runtime I/O, one MCM control, build work that reuses tables already parsed. Convention needed: which `$keys` the generator bakes (naming rule or explicit list).

Player cost of wrong language is a wrong string once per rare event - not save risk, not frame cost.

---

## Settled (stop re-litigating)

| Topic | Settlement |
|-------|------------|
| `$key` into `Debug.Notification` | Does not localize. Dead end. |
| Runtime parse of translation files | Not required; bake is the path. |
| Auto-detect `sLanguage` | Wrong for this mod's menu-language model. |
| Manual enum without seed | Still shippable; seed is polish for zero-click. |
| Seed without re-detecting "Chinese players" | Correct; follow installer menu language. |
| Punchline English | Taste, not a technical ceiling. |
| New key on upgrade | Works when key is only in Config; user Settings omit it until changed. |
